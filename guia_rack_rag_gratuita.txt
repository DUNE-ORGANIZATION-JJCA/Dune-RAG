Guía de implementación
Rack RAG gratuito y robusto
Selección técnica, arquitectura, despliegue y especificaciones
para un sistema de recuperación y generación aumentada fácil de montar

Stack elegido: Qwen + Ollama + Haystack + Qdrant + Docling + Crawl4AI + SBERT + FastAPI + Frontend + Ragas

Documento de trabajo preparado para despliegue inicial, validación técnica y evolución posterior.


1. Objetivo del documento
Este documento define un rack RAG gratuito, claro de desplegar y suficientemente sólido para pasar de una prueba funcional a una base seria de trabajo.
La prioridad no es meter todas las piezas posibles, sino elegir las que aportan más valor con menos fricción operativa: extracción fiable, indexación robusta, consultas consistentes, API limpia y una ruta sencilla para escalar después.
La selección está pensada para un primer entorno local o de laboratorio y para un segundo paso en servidor o VPS con mínimas modificaciones.

3. Rack seleccionado y criterio de elección
3.1 LLM principal: Qwen
Se recomienda Qwen como modelo base del sistema.
Encaja bien cuando se busca un modelo capaz de trabajar en local, responder en espańol con soltura y servir tanto para consultas simples como para flujos con herramientas.
Para empezar: usar una versión ligera en desarrollo y una más capaz cuando ya esté todo estable.
3.2 Runtime del modelo: Ollama
Ollama se utiliza como capa de ejecución del modelo.
Su valor está en que simplifica muchísimo el arranque: bajar el modelo, levantar el servicio y consumirlo desde una API muy fácil de integrar.
Reduce el tiempo de montaje y evita pelear con despliegues manuales del servidor del modelo.
3.3 Framework de orquestación: Haystack
Haystack será el cerebro del flujo RAG.
Se elige porque permite construir pipelines limpios: consulta, recuperación, ranking, generación y posprocesado sin convertir el proyecto en un conjunto de scripts desordenados.
Es una buena base para dejar el sistema entendible también para terceros.
3.4 Vector database: Qdrant
Qdrant será la base donde quedarán indexados los embeddings y los metadatos de los documentos.
Su punto fuerte es que sirve bien tanto en entornos pequeńos como en despliegues más serios y permite crecer con filtros, payloads y búsquedas híbridas.
Es mejor base de largo recorrido que opciones más orientadas a prototipos rápidos.
3.5 Extracción de documentos: Docling
Docling será el extractor principal para PDF, DOCX, PPTX y otros formatos documentales.
La razón es simple: cuanto mejor entra el documento, mejor chunking y mejor recuperación tendrás después.
El sistema debe limpiar y estructurar bien antes de indexar, no solo partir texto sin criterio.
3.6 Extracción web: Crawl4AI
Para URLs y páginas web se usará Crawl4AI.
Separar la ingesta web de la documental evita mezclar flujos distintos y mejora la limpieza del contenido recuperado.
Esto reduce ruido y mejora la calidad de las respuestas.
3.7 Embeddings: SBERT
SBERT será la capa de embeddings inicial.
Ofrece una combinación muy buena entre coste cero, velocidad y calidad suficiente para un RAG robusto de primera fase.
Se puede empezar con un modelo ligero y pasar a otro más fuerte sin rediseńar el proyecto.
3.8 API, frontend y evaluación
FastAPI se usará para exponer la API del sistema.
React es la opción recomendada si el objetivo es una interfaz más profesional y escalable; Streamlit si se prioriza una consola funcional muy rápida de poner en marcha.
Ragas se incorporará cuando el sistema ya responda, para medir si recupera bien y si realmente contesta apoyándose en el contexto.
4. Arquitectura funcional recomendada
El flujo general debe quedar así:
1. El usuario consulta desde la interfaz.
2. La petición entra por FastAPI.
3. Haystack gestiona el pipeline.
4. Se ejecuta la recuperación en Qdrant usando embeddings SBERT.
5. Se selecciona el contexto útil.
6. Qwen, servido por Ollama, genera la respuesta final.
7. La respuesta vuelve con referencias a los documentos recuperados.
Consulta: Frontend ? FastAPI ? Haystack ? Qdrant + Ollama/Qwen
Ingesta: Documentos y URLs ? Docling / Crawl4AI ? limpieza y chunking ? SBERT ? Qdrant
Evaluación: Dataset interno ? Ragas ? ajustes del pipeline
Este diseńo mantiene separadas tres capas: ingesta, consulta y evaluación. Esa separación es lo que da robustez desde el principio.

7. Fases de implementación
Fase 1. Levantar la base
* Crear el repositorio y la estructura de carpetas.
* Levantar Qdrant y Ollama con Docker.
* Comprobar que ambos servicios responden antes de escribir lógica adicional.
Fase 2. Preparar el modelo
* Descargar el modelo Qwen elegido.
* Realizar consultas directas al modelo para verificar que responde correctamente.
* No avanzar al RAG hasta confirmar que el modelo base está estable.
Fase 3. Construir la API
* Crear FastAPI con endpoints mínimos: /health, /ingest y /query.
* Separar rutas, servicios y pipelines desde el principio.
* Ańadir logs sencillos y mensajes de error claros.
Fase 4. Construir la ingesta
* Procesar documentos con Docling.
* Procesar URLs con Crawl4AI.
* Limpiar texto, trocear contenido y generar embeddings con SBERT.
* Guardar chunks y metadatos en Qdrant.
Fase 5. Conectar el pipeline RAG
* Recibir consulta del usuario.
* Crear embedding de la consulta.
* Recuperar contexto en Qdrant.
* Enviar contexto al modelo Qwen a través de Ollama.
* Devolver respuesta con trazabilidad básica.
Fase 6. Medir y corregir
* Crear un pequeńo conjunto de preguntas reales.
* Evaluar con Ragas.
* Ajustar chunking, prompts y recuperación antes de ańadir complejidad extra.


8. Implementación sencilla de servicios
A nivel práctico, la primera versión puede montarse con un docker-compose muy corto y limpio. La lógica de aplicación irá fuera o en contenedores propios según prefieras.
services:
  qdrant:
    image: qdrant/qdrant
    ports:
      - "6333:6333"
    volumes:
      - ./volumes/qdrant:/qdrant/storage

  ollama:
    image: ollama/ollama
    ports:
      - "11434:11434"
    volumes:
      - ./volumes/ollama:/root/.ollama

  api:
    build: ./api
    ports:
      - "8000:8000"
    depends_on:
      - qdrant
      - ollama

  frontend:
    build: ./frontend
    ports:
      - "3000:3000"
    depends_on:
      - api
Con esto ya puedes levantar el esqueleto. Después ańades el worker de ingesta como servicio aparte o lo ejecutas manualmente durante la primera fase.

Criterios de robustez desde el día 1
* Separar ingesta y consulta en módulos distintos.
* Persistir Qdrant y Ollama en volúmenes propios.
* Mantener variables sensibles en .env y no en el código.
* Ańadir endpoint /health para revisar disponibilidad del sistema.
* Guardar metadatos útiles por chunk: fuente, tipo, ruta, fecha y sección.
* Registrar errores de parsing y documentos rechazados.
* No mezclar extracción, chunking y respuesta final en una sola función gigante.
11. Checklist de validación
* Qdrant responde y permite crear colección.
* Ollama responde y tiene el modelo descargado.
* La API levanta y devuelve estado correcto en /health.
* Un documento de prueba se ingesta sin errores.
* Una consulta sencilla devuelve contexto coherente.
* La respuesta final menciona información realmente presente en los chunks recuperados.
* Ragas puede ejecutarse con un pequeńo dataset interno.
12. Qué no ańadir todavía
* Memoria conversacional compleja si aún no has validado el retrieval.
* Multiagente si todavía no tienes claro el flujo básico.
* Cachés semánticas antes de medir si realmente hacen falta.
* Demasiadas fuentes de datos en la primera iteración.
* Rerankers y capas extra de scoring sin una razón observada.
13. Ruta de evolución recomendada
Primero: base funcional con consulta e ingesta estables.
Después: mejorar recuperación, trazabilidad y evaluación.
Luego: ańadir reranking, filtros avanzados, autenticación y observabilidad.
Al final: escalar a varios repositorios, fuentes o agentes especializados si el caso lo justifica.
14. Conclusión
La combinación recomendada —Qwen, Ollama, Haystack, Qdrant, Docling, Crawl4AI, SBERT, FastAPI y Ragas— es una base muy equilibrada para montar un RAG gratuito, útil y mantenible. No busca impresionar con complejidad, sino ofrecer una estructura sólida que puedas desplegar, probar y mejorar sin rehacer el proyecto a mitad de camino.
La clave del éxito no estará en meter más piezas, sino en dejar muy bien resueltas estas cinco: ingesta, chunking, embeddings, recuperación y generación.
